    Semaphore Runner Design（Runner 構成ガイド）¶

    このドキュメントでは、Semaphore の Runner の構成・動作・運用ルールをまとめる。
    本環境は Docker Compose + 内部 Runner モード（server --runner） を前提とする。

    🎯 Runner の役割¶

    Semaphore Runner は、以下の処理を担当する実行エンジンである：

    • Ansible Playbook の実行
    • SSH 接続（管理対象ノードへのアクセス）
    • ログ収集と Semaphore への送信
    • GitOps 実行時の Playbook 実行処理

    Runner が正常に動作しないと、Semaphore は Playbook を実行できない。

    🧩 1. Runner の種類¶

    Semaphore には 2 種類の Runner が存在する：

    種類 説明 特徴
    内部 Runner（Internal Runner） Semaphore 本体に内蔵 設定が簡単・トークン不要
    外部 Runner（External Runner） 別コンテナとして起動 大規模環境向け・冗長化可能

    本環境では 内部 Runner を採用している。

    🧩 2. 内部 Runner の構成（本環境）¶

    ✔ docker-compose.yml の設定¶

    Semaphore コンテナの起動コマンドに --runner を付与する：

    command: server --config /etc/semaphore/config.json --runner

    これにより、Semaphore 本体が Runner を内蔵して起動する。

    ✔ 内部 Runner の特徴¶

    • Runner の登録が不要（トークン不要）
    • Runner の設定ファイルも不要
    • 自動的に Semaphore に紐づく
    • 小規模〜中規模環境に最適

    🧩 3. Runner の動作フロー¶

    GitHub → Semaphore → Runner → Managed Nodes
    1. GitHub から最新の Playbook を pull 2. Semaphore がテンプレートを実行 3. Runner が Playbook を実行 4. SSH 鍵を使って管理対象ノードに接続 5. ログを Semaphore に送信 6. GUI にログが表示される

    4. Runner の SSH 鍵¶

    Runner は以下の鍵を使用して管理対象ノードに接続する： 

    /etc/semaphore/keys/mike_key

    Environment で以下のように設定する： 

    ansible_ssh_private_key_file=/etc/semaphore/keys/mike_key

    ✔ 鍵の権限（最重要）¶

    ホスト側： 

    chown 1001:0 /opt/semaphore/keys
chmod 700 /opt/semaphore/keys
    これが正しくないと Runner が鍵を読み取れず、SSH 接続に失敗する。

    🧩 5. Runner のログ¶

    Runner のログは Semaphore GUI から確認できる： - テンプレート実行ログ - SSH 接続ログ - Ansible の stdout / stderr - 成功 / 失敗ステータス

    ログはコンテナ内にも保存される： 

    /var/lib/semaphore/logs/

    🧩 6. Runner のトラブルシュート¶

    症状 原因 対処
    Playbook が実行されない Runner が起動していない dockerlogs semaphore を確認
    SSH 接続に失敗 鍵の権限不一致 /opt/semaphore/keys の権限を修正
    Git pull に失敗 Deploy Key の権限不足 GitHub の設定を確認
    Runner not found 外部 Runner 用設定を誤って使用 内部 Runner モードを確認
    ログが空 Runner が落ちている コンテナ再起動

    🧠 ベストプラクティス¶

    • 小規模環境では 内部 Runner が最適
    • 鍵は Key Store にのみ保存し、ローカルに置かない
    • Runner のログは定期的にバックアップ
    • SSH 鍵は ed25519 を推奨
    • Playbook は冪等性を維持し、何度実行しても安全な状態にする
    • Runner の負荷が増えたら外部 Runner への移行を検討

    🧩 7. 外部 Runner への拡張（将来）¶

    大規模環境では外部 Runner を複数台構成する： 

    docker run -d \
  -e SEMAPHORE_URL=https://semaphore.example.com \
  -e SEMAPHORE_RUNNER_TOKEN=xxxx \
  semaphoreui/runner:latest

    • 冗長化
    • 負荷分散
    • 大規模 Playbook の高速化 本環境ではまだ不要だが、将来的に拡張可能。

    ✅ まとめ¶

    Semaphore Runner は GitOps の実行エンジンであり： - Playbook 実行 - SSH 接続 - ログ収集 - GitHub pull の反映

    すべての中心となる。 内部 Runner モードにより、シンプルで安定した GitOps 基盤が実現している。
